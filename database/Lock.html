# 锁

# 用于解决并发事务带来的问题

  • 读-读: 情况没有资源竞争不需要锁
  • 写-写: 会发生脏写问题
    • 当Mysql要查询某一条记录时候,就形成该记录的锁结构,分别记录事务 trx_id 和 is_waiting 当前事务状态
  • 读-写: 会发生脏读\幻读\不可重复读的情况
    • 处理方案: MVCC多版本控制

# 写操作

  • Delete:

从B+树中定位到数据的位置，获取X锁，对该记录做Delete Mark操作

  • UPDATE:
    • 如果修改的数据空间不变，与删除操作类似
    • 如果修改的数据空间一样，先彻底删除数据(把记录移动到垃圾链表中), 再插入一条记录, X锁和锁定读, 隐式锁保护新插入的数据
    • 如果修改了键值则先删除数据，后插入数据,按照DELETE和INSERT规则就可

# 多粒度锁

添加意向共享锁和意向独占锁来为添加表级锁服务

  • 表级别S锁, X锁

# InnoDB中的行级锁

  • Record Locks:

LOCK_REC_NOT_GAP, 锁住当前一整条记录

  • Gap Locks:

不允许在 Gap Locks　前面的间隙插入数据

  • Next-Key Locks:

是 Record Lock 和 Gap Locks的组合

  • Insert Intention Locks:

存储要向记录加上的锁

  • 隐式锁
    • 对于聚簇索引中，有隐藏列trx_id记录了每一个修改索引的trx_id，当要对记录加锁时候，则查看当前进行的trx_id是否该隐藏列中
